> 日期：2026-07-21  
> 状态：设计已确认，待实施

## 1. 目标

在 MrFung 网站发布一篇中文技术白皮书型长文，系统介绍 Kotlin Multiplatform 在 iOS、Android 与 HarmonyOS 高频数据场景中的实测结果、边界设计、内存治理方法和工程选型结论。

文章必须满足以下目标：

- 只保留可公开的通用技术内容，不出现具体产品、证券品类、内部项目编号、内部工程名、内部类名、故障排查过程或本地日志路径。
- 以同一份测试负载和真实设备数据对比原生实现、标准 KMP 接入和高频边界优化方案。
- 不用单个平台的绝对耗时与另一平台横向排名，而是在同一设备、同一负载、同一构建配置内比较方案。
- 每张图表同时给出数据、含义和工程判断，使读者能区分“运行速度快”“内存基线低”“长时间稳定”三类不同结论。
- 文章无需联网加载图表库、字体或数据文件，复制网站静态文件后仍可完整阅读。
- 文章加入现有文章目录、首页入口、站点地图和订阅源，并沿用网站的 12 种语言外壳。

- 通用架构、边界优化、内存治理、生命周期和稳定性验收原则。

### 3.2 排除内容

- 具体业务名称、金融或证券品类、具体公司或产品名。

### 3.3 场景抽象

将测试对象统一描述为“5000 行 × 50 字段的实时数据表”，包含：

- 首次装载与批量状态计算；
- 每秒约 20 批的常态增量输入；
- 每秒约 100 批的突发增量输入；
- 34 行可视区域和预取窗口；
- 每批最高 816 个字段变化；
- 增量数据合并、排序/筛选状态更新和 UI Patch 输出。

文章说明该负载用于放大高频跨语言边界的分配、拷贝、对象图和队列问题，不对应某一项具体业务。

## 4. 文章结构

### 4.1 开篇结论

首屏直接给出三个结论卡片：

1. 一份 KMP Core 源码可以覆盖三端，但三端不应强行使用同一种调用边界。
2. Android 直接使用 KMP/JVM；iOS 普通接口使用 Kotlin/Native Framework、高频批量接口使用同一 Core 的 C ABI；HarmonyOS 高频链路使用 CPF-KMP + AKInterop 的直接缓冲区方案。
3. KMP 不是天然内存泄漏源，真正风险来自无界队列、完整快照、跨边界对象图、重复拷贝和生命周期未释放；内存验收必须包含长时间真实速率测试。

### 4.2 测试方法

说明以下可比性约束：

- 三种实现消费同一二进制场景和命令序列。
- 用最终状态摘要校验语义等价，性能快但结果不一致的样本视为无效。
- 真机 Release 构建；短场景采用多轮交错运行，长稳场景按真实时间持续输入。
- 延迟同时展示中位数、百分之九十五位置和百分之九十九位置，但正文使用中文名称解释，不用缩写作为主要表达。
- 内存区分测试前、测试后、最高点、结束恢复值和净增长，避免只截取单点。
- 不跨平台比较绝对值，只在各平台内部比较方案。

### 4.3 共享 Core 与平台边界

用一张架构图表达：

```text
同一 Kotlin commonMain Core
        │
        ├── Android：KMP/JVM 直接调用
        ├── iOS：Framework 普通接口 + C ABI 高频批量接口
        └── HarmonyOS：CPF-KMP Kotlin/Native + AKInterop/N-API 直接缓冲区
```

Core 负责数据模型、合并、计算、排序筛选状态和平台无关 Patch；网络连接、线程调度、页面生命周期、UI 渲染、崩溃采集与平台资源由各端 Adapter 负责。

### 4.4 iOS 实测与选型

对比同一 iPhone、同一场景下的 Swift 原生、官方 Kotlin/Native Framework 和同一 KMP Core 的 C ABI。

重点图表：

- 常态与突发增量的尾部延迟；
- 每秒可处理批次数；
- 最高常驻内存；
- Framework 与 C ABI 的适用边界。

文章结论：官方 Framework 的绝对性能已满足常见频率，但高频对象通道使用 C ABI 批量缓冲区可进一步降低尾部延迟和内存；这不是更换 KMP 运行时，而是为同一 Core 增加窄而稳定的高频出口。

### 4.5 Android 实测与选型

对比同一 Android 真机、同一场景下的原生 Kotlin 与 KMP/JVM。

重点图表：

- 常态与突发增量尾部延迟；
- 最高 PSS；
- 延迟与内存门槛是否通过。

文章结论：Android 不需要额外桥接层，KMP/JVM 与原生 Kotlin 处于同一运行时，直接调用共享 Core 的复杂度和维护成本最低。

### 4.6 HarmonyOS 实测与选型

对比 ArkTS 原生、KuiklyBase/KNOI 旧接入和 CPF-KMP + AKInterop 新接入。

重点图表：

- 三方案短测最高 PSS；
- KNOI 优化前后内存；
- CPF-KMP 约 31 分钟长稳测试的最高值、结束恢复值和队列积压；
- CPF-KMP 相对优化后 KNOI、ArkTS 原生的内存差值。

文章明确新方案使用 Kotlin/Native，不使用 Kotlin/JS。性能改善来自更短的调用链、同步借用 ArrayBuffer、扁平二进制数据和避免大规模 Kotlin 对象图跨过 ArkTS/N-API 边界，而不是编译器自动消除所有内存问题。

### 4.7 高频数据的内存治理

整理为可复用的工程清单：

- 输入队列必须有界；消费落后时采用最新值覆盖、按键合并或明确丢弃策略。
- 跨边界传扁平批量数据，不逐字段创建语言间对象图。
- 调用方预分配或复用缓冲区；同步调用期间借用，返回后不保存外部指针。
- Core 输出增量 Patch，不重复构造完整数据快照。
- UI 合并刷新且只保留最新待渲染 Patch，不让 UI 速度反向撑大 Core 队列。
- 退出页面、取消订阅、切换数据源和进入后台时有明确释放语义。
- 短测看吞吐和尾部延迟，长测看内存斜率、最高点、恢复值和积压量。

### 4.8 总体选型矩阵

用表格给出最终落地建议：

| 平台 | 默认通道 | 高频批量通道 | 选型 |
| --- | --- | --- | --- |
| Android | KMP/JVM 直接 API | 同一直接 API + 批量结构 | 直接使用共享 Core |
| iOS | 官方 Kotlin/Native Framework | 同一 Core 的 C ABI + Swift Adapter | 双通道 |
| HarmonyOS | CPF-KMP 普通接口 | AKInterop/N-API Direct ArrayBuffer | CPF-KMP 新通道 |

最终强调“一套源码”指业务 Core 的单一事实源，不等同于三端必须共享完全相同的 ABI、内存表示和 UI 调度机制。

### 4.9 限制与复现条件

- 真机样本对应特定设备与系统版本，升级编译器、运行时、系统或桥接工具后应重跑基准。
- 数据说明方案在本场景和负载范围内通过，不证明所有 KMP 代码天然高效。
- 长稳内存需要结合系统 PSS/常驻内存、队列深度和 Core 内部对象计数共同判断。
- CPF-KMP 与 AKInterop 采用锁定版本和提交；公开仓库、许可证和持续维护状态应在正式采用前再次核验。

## 5. 图表与数据设计

### 5.1 数据展示原则

- 图表使用站内 HTML/CSS 与内联 SVG，不引入 ECharts、Chart.js、CDN 或运行时请求。
- 每张图有可见标题、单位、图例、精确数值标签和“性能分析”段落。
- 同一颜色在全文保持同一方案语义；不只靠颜色区分，柱形或折线同时带文字标签。
- 图表下方提供语义表格，既作为精确值来源，也便于屏幕阅读器读取。
- 移动端图表允许内容重排，不允许页面产生横向滚动。
